# Chapter 1: Introduction

## 1.1 Project Overview

**CodeSync** is a full-stack, real-time collaborative Integrated Development Environment (IDE) that enables multiple developers to write, edit, and manage code simultaneously within shared workspaces. The application is designed to bring the collaborative power of tools like Google Docs to the world of software development.

At its core, CodeSync provides:
- A professional-grade **Monaco Editor** (the same engine behind VS Code)
- **Role-Based Access Control (RBAC)** with Owner / Editor / Viewer roles
- **Live cursor tracking** and user presence indicators
- **File version history** with restore capability
- **Real-time activity feeds** and member management
- **Instant file synchronization** across all clients via Socket.IO

## 1.2 Motivation

In modern software engineering, distributed teams are the norm. Developers collaborate across time zones, requiring tools that support seamless remote pair programming, code reviews, and live debugging sessions. Existing solutions are either too heavyweight (full cloud IDEs) or too simple (basic shared editors). CodeSync fills this gap by delivering a lightweight, installable, full-featured collaborative IDE.

## 1.3 Objectives

- Build a production-grade collaborative code editor accessible from any browser
- Implement real-time bi-directional communication using WebSockets
- Provide a structured permission model to control who can read, write, or manage workspaces
- Deliver a premium, modern dark-mode UI with smooth animations and UX micro-interactions
- Ensure performance through debouncing, throttling, lazy loading, and database indexing

## 1.4 Scope

| In Scope | Out of Scope |
|----------|-------------|
| Real-time code collaboration | In-browser code execution (planned) |
| File & folder management | Mobile native apps |
| Member management & RBAC | Offline support |
| Version history & restore | Conflict resolution via OT/CRDT (planned) |
| Activity logging | Email notifications (planned) |
| JWT authentication | OAuth2/SSO (planned) |

## 1.5 Project Version

**CodeSync v3.0 Turbo** — February 2026

# Chapter 2: Literature Survey

## 2.1 Existing Collaborative Tools

### 2.1.1 Google Docs / Collaborative Text Editors
Google Docs pioneered real-time collaborative editing using **Operational Transformation (OT)**. While excellent for documents, it lacks code-specific features like syntax highlighting, language templates, and developer tooling.

### 2.1.2 Visual Studio Code Live Share
Microsoft's Live Share extension for VS Code allows real-time code collaboration. However, it requires a local VS Code installation, is platform-dependent, and requires port forwarding or cloud relay — increasing setup complexity.

### 2.1.3 CodePen / JSFiddle / Replit
These platforms offer online code editing, some with collaboration features. Replit has real-time collaboration but includes an execution engine, making it heavier. These tools lack fine-grained RBAC, ownership transfer, or advanced member management.

### 2.1.4 GitHub Codespaces
A full cloud IDE backed by VS Code. Powerful but resource-intensive, requires GitHub subscription, and has significant cloud compute costs. Not suitable for lightweight team collaboration on a budget.

## 2.2 Technology Research

### 2.2.1 WebSockets & Socket.IO
WebSockets provide full-duplex communication over a single TCP connection. Socket.IO abstracts WebSocket with:
- Automatic reconnection
- Room-based namespacing
- Fallback to HTTP long-polling

Socket.IO was chosen over raw WebSockets and alternatives (SignalR, Pusher) for its maturity, Node.js ecosystem fit, and built-in room support.

### 2.2.2 Monaco Editor
The Monaco Editor is the code editor that powers VS Code, extracted as a standalone browser library. It provides:
- Syntax highlighting for 50+ languages
- IntelliSense / autocomplete
- Diff editing
- Minimap, bracket pairing, multi-cursor support

### 2.2.3 Operational Transformation vs CRDT
Real-time collaborative editors must resolve conflicts when two users edit the same position simultaneously.
- **OT (Operational Transformation)**: Used by Google Docs. Transforms operations to account for concurrent edits.
- **CRDT (Conflict-free Replicated Data Types)**: Used by Yjs, Automerge. Mathematically guaranteed conflict-free merges.

CodeSync v3.0 uses a **last-write-wins with debounce** model for simplicity. OT/CRDT is planned for a future release.

### 2.2.4 JWT Authentication
JSON Web Tokens (JWT) provide stateless, server-scalable authentication. The server signs a token with a secret; clients include it in Authorization headers. No session storage is needed on the server.

## 2.3 Research Gap

No existing lightweight solution combines:
1. A VS Code-grade editor in the browser
2. Fine-grained RBAC (Owner/Editor/Viewer)
3. Hierarchical file system with multi-level folders
4. Version history with restore
5. Real-time presence, cursor sync, and activity feed

**CodeSync addresses this gap.**

# Chapter 5: Algorithm Design

## 5.1 Real-Time Code Synchronization

### 5.1.1 Debounced Auto-Save Algorithm

```
ON code_change event received at server:
  1. Extract { fileId, code, userId } from socket payload
  2. Check if userId has role >= EDITOR in workspace (RBAC guard)
  3. Broadcast { fileId, code } to all sockets in room `file:${fileId}` EXCEPT sender
  4. Clear existing debounce timer for fileId (if any)
  5. Set new debounce timer (5000ms delay):
       → ON timer fires: UPDATE File.content WHERE _id = fileId
  6. Return (no immediate DB write)

RESULT: DB writes occur at most once per 5 seconds per file, regardless of typing speed.
```

### 5.1.2 Cursor Throttle Algorithm

```
ON cursor_position received at server:
  1. Extract { fileId, position, username, color }
  2. Broadcast cursor_update to all sockets in `file:${fileId}` EXCEPT sender
  (Throttling is handled client-side before emitting)
```

## 5.2 File Name Normalization Algorithm

When creating or renaming a file/folder:

```
INPUT: rawName (user-typed string)

STEP 1: Trim whitespace from both ends
          "  my file  " → "my file"

STEP 2: Strip special characters (keep alphanumeric, dots, hyphens, underscores, spaces)
          "my file!@#$" → "my file"

STEP 3: Collapse multiple spaces into one
          "my   file" → "my file"

STEP 4: Convert to lowercase for uniqueness check (store as nameLower)
          "My File" → "my file"

STEP 5: Check compound unique index (workspaceId + path + nameLower)
          → If exists: return 409 Conflict
          → If not: proceed with creation/rename

STEP 6: Persist both name (display) and nameLower (comparison) to MongoDB
```

## 5.3 Recursive Folder Delete Algorithm

```
DELETE_RECURSIVE(folderId, workspaceId):

  1. Find folder document by folderId
  2. Compute folderPath = folder.path + "/" + folder.name
  3. Delete all File documents WHERE:
       workspaceId = workspaceId
       AND (path = folderPath OR path starts with folderPath + "/")
  4. Delete all FileVersion documents for each deleted file
  5. Emit socket event: folder_children_deleted { parentPath }
  6. Delete the folder document itself
  7. Emit socket event: file_deleted { fileId }
```

## 5.4 Recursive Rename Algorithm

When a folder is renamed, all children must have their path updated:

```
RENAME_FOLDER(folderId, newName):

  1. Compute oldPath = folder.path + "/" + folder.name
  2. Compute newPath = folder.path + "/" + newName
  3. Update all File documents WHERE path starts with oldPath:
       Replace prefix oldPath with newPath in each document's path field
  4. Update folder.name → newName
  5. Emit: file_renamed { fileId, name, oldName, path }
```

## 5.5 Ownership Transfer Algorithm

```
TRANSFER_OWNERSHIP(workspaceId, currentOwnerId, targetUserId):

  1. Verify requestor is current owner (RBAC check)
  2. Find WorkspaceMember for currentOwner → set role = 'editor'
  3. Find WorkspaceMember for targetUser → set role = 'owner'
  4. Update Workspace.owner = targetUserId
  5. Log activity: OWNERSHIP_TRANSFERRED
  6. Return updated member list
```

## 5.6 User Cascade Delete Algorithm

When a user account is deleted:

```
DELETE_USER_CASCADE(userId):

  1. Find all Workspaces owned by userId
  2. For each owned workspace:
     a. Delete all Files in workspace
     b. Delete all FileVersions for those files
     c. Delete all WorkspaceMembers in workspace
     d. Delete all ActivityLogs for workspace
     e. Delete the Workspace document
  3. Delete all WorkspaceMember documents where userId = userId (non-owned)
  4. Delete the User document
```

# Chapter 6: Database Design

## 6.1 Database Technology

**MongoDB** (via **Mongoose 8**) was chosen for its:
- **Schema flexibility** — suitable for evolving project data shapes
- **Document model** — natural fit for nested file/folder structures
- **Atlas cloud** — managed hosting with free tier
- **Compound indexes** — efficient multi-field unique constraints

## 6.2 Entity Relationship Overview

```
User ────────────────────────────── WorkspaceMember
 │                                        │
 │                                        │
 └──────── Workspace ─────────────────────┘
               │
               ├─── File ─────── FileVersion
               │
               └─── ActivityLog
```

## 6.3 Schema Definitions

### 6.3.1 User Model

| Field | Type | Constraints |
|-------|------|-------------|
| `username` | String | Required, trimmed |
| `email` | String | Required, unique, lowercase |
| `password` | String | bcrypt hashed (10 rounds) |
| `createdAt` | Date | Auto (timestamps) |
| `updatedAt` | Date | Auto (timestamps) |

### 6.3.2 Workspace Model

| Field | Type | Constraints |
|-------|------|-------------|
| `name` | String | Required, max 80 chars |
| `owner` | ObjectId → User | Required |
| `language` | Enum | `javascript`, `typescript`, `python`, `java`, `cpp`, `go`, `rust`, `html`, `css`, `json`, `txt` |
| `createdAt` | Date | Auto |

### 6.3.3 WorkspaceMember Model

| Field | Type | Constraints |
|-------|------|-------------|
| `workspaceId` | ObjectId → Workspace | Required |
| `userId` | ObjectId → User | Required |
| `role` | Enum | `owner` / `editor` / `viewer` |
| `invitedBy` | ObjectId → User | Optional |
| **Index** | Compound Unique | `(workspaceId, userId)` |

### 6.3.4 File Model

| Field | Type | Constraints |
|-------|------|-------------|
| `workspaceId` | ObjectId → Workspace | Required |
| `name` | String | Required, trimmed |
| `nameLower` | String | Auto pre-save (lowercase of name) |
| `path` | String | Directory path, default `/` |
| `type` | Enum | `file` / `folder` |
| `content` | String | File contents (empty for folders) |
| `language` | String | Auto-detected from file extension |
| `createdBy` | ObjectId → User | — |
| `lastEditedBy` | ObjectId → User | — |
| **Index** | Compound Unique | `(workspaceId, path, nameLower)` |

### 6.3.5 FileVersion Model

| Field | Type | Constraints |
|-------|------|-------------|
| `fileId` | ObjectId → File | Required |
| `content` | String | Snapshot of file content |
| `editedBy` | ObjectId → User | Required |
| `createdAt` | Date | Auto |

### 6.3.6 ActivityLog Model

| Field | Type | Constraints |
|-------|------|-------------|
| `workspaceId` | ObjectId → Workspace | Required, indexed |
| `userId` | ObjectId → User | Required |
| `actionType` | String | e.g. `USER_JOINED`, `FILE_CREATED`, `ROLE_CHANGED` |
| `targetId` | ObjectId | Optional target entity |
| `metadata` | Mixed | Additional context (e.g. old/new role) |
| `createdAt` | Date | Auto |

## 6.4 Indexes Summary

| Model | Index Type | Fields | Purpose |
|-------|-----------|--------|---------|
| User | Unique | `email` | Fast login lookup, prevent duplicates |
| WorkspaceMember | Compound Unique | `(workspaceId, userId)` | One membership per user per workspace |
| File | Compound Unique | `(workspaceId, path, nameLower)` | Case-insensitive uniqueness within folder |
| ActivityLog | Standard | `workspaceId` | Fast activity feed queries |

# Chapter 8: Security Design

## 8.1 Authentication

### JWT-Based Stateless Auth

1. **Registration**: Password hashed with **bcrypt** (10 salt rounds) before storage
2. **Login**: Email looked up → bcrypt.compare → if valid, JWT signed with `JWT_SECRET`
3. **Protected Routes**: Every request to protected endpoints must include `Authorization: Bearer <token>`
4. **Middleware**: `authMiddleware.js` verifies token signature and expiry on every request

```
Client Login Request
        │
        ▼
  bcrypt.compare(password, hash)
        │ ✅ Match
        ▼
  jwt.sign({ userId }, JWT_SECRET, { expiresIn: '7d' })
        │
        ▼
  Return { token, user } to client

Subsequent Requests:
  Header: Authorization: Bearer <token>
        │
        ▼
  jwt.verify(token, JWT_SECRET)
        │ ✅ Valid
        ▼
  req.user = decoded payload → proceed
```

## 8.2 Authorization (RBAC)

Three middleware functions protect routes:

| Middleware | Function |
|-----------|----------|
| `authMiddleware` | Verifies JWT is present and valid |
| `workspaceAuth` | Confirms user is a member of the workspace |
| `permissionMiddleware.requireRole(roles)` | Confirms user's role meets minimum required |

Example route protection chain:
```
router.delete('/:id', authMiddleware, workspaceAuth, requireRole(['owner']), deleteWorkspace)
```

## 8.3 Socket-Level Security

Viewer role enforcement is applied at the **socket server level**, not just the UI:

```javascript
// On code_change event:
if (userRole === 'viewer') {
  // Silently drop the event — do not broadcast or save
  return;
}
```

This prevents malicious clients from bypassing the UI to send unauthorized edits.

## 8.4 Input Validation & Sanitization

- File names are sanitized using a normalization algorithm (Chapter 5.2)
- Compound unique index prevents duplicate file names at DB level
- Mongoose schema validation enforces required fields and enum values
- Express `express.json()` with size limits prevents payload bloat

## 8.5 CORS Configuration

CORS is configured to only allow requests from `FRONTEND_URL` (set in `.env`), blocking cross-origin requests from unauthorized sources.

## 8.6 Security Headers (Recommended for Production)

| Header | Purpose |
|--------|---------|
| `helmet.js` | Sets security headers (CSP, X-Frame-Options, etc.) |
| `rate-limiter-flexible` | Prevent brute-force attacks on `/api/auth/login` |
| HTTPS/WSS | Encrypt all traffic in production |

## 10.3 Discussion

### What Worked Well

1. **Socket.IO rooms** proved an excellent match for workspace + file-level isolation of events
2. **Debounced auto-save** dramatically reduces MongoDB write load while maintaining good UX
3. **Compound unique index** on `(workspaceId, path, nameLower)` elegantly solves case-insensitive file name conflicts at the DB level
4. **Mongoose pre-save hooks** to auto-generate `nameLower` is clean and transparent
5. **React lazy loading with Vite** delivers fast page loads even with a large codebase

### Challenges Encountered

1. **Recursive rename** — updating all children paths when a folder is renamed requires careful MongoDB query design using `$regex` prefix matching
2. **Socket disconnect cleanup** — tracking active users across multiple workspaces/files requires careful in-memory state management on the server
3. **Viewer enforcement** — had to apply checks at both the UI level (disable editor) AND socket server level (drop events) to prevent bypasses
4. **Real-time consistency** — without OT/CRDT, simultaneous edits by two users can overwrite each other; mitigated by debounce but not fully solved
